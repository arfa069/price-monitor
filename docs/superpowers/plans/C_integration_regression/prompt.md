# 提示词C（V2）：联调、回归与边界验收

你是测试负责人 + 全栈工程师。请对当前项目执行“前后端联调 + 回归 + 边界测试”，并输出结构化验收报告。

一、测试范围（必须覆盖）
1) 前端基础行为
- 访问 http://localhost:3000 默认进入商品管理页
- 左侧边栏常驻不消失，包含“商品管理”“定时配置”，点击后右侧路由切换
- 非法路径自动重定向到商品管理页

2) 商品管理（服务端分页）
- 列表查询、筛选、关键词搜索可用
- 分页每页固定 15 条
- 单条新增/编辑/删除
- 批量新增（多 URL 粘贴）
- 批量删除、批量启停
- 分页与筛选组合下行为正确

3) 定时配置（真实后端）
- 页面加载 GET /config 展示真实 crawl_cron
- PATCH /config 保存 crawl_cron 成功后页面同步
- localStorage 仅作为草稿兜底
- “草稿与后端不一致”时，能恢复草稿或丢弃草稿

4) 后端调度
- APScheduler 启动后按 crawl_cron 调度
- 更新 crawl_cron 后无需重启即生效
- 若上一次任务未完成，本次触发跳过且有日志
- 时区按 Asia/Shanghai 运行

5) 回归
- POST /crawl/crawl-now 仍可手动触发
- /health、/products、/config 无回归
- pytest 全量通过

二、边界场景（必须单独验收）
1) /products page 越界时返回空 items + 正常分页元信息（非 500）
2) 筛选无结果时 total=0 且页面无报错
3) 批量新增去重：
- 输入内重复 URL 去重
- 与已存在商品重复时给出明确反馈
4) 批量操作部分失败时，显示成功/失败数量与失败原因
5) 删除后当前页为空时，前端自动回退上一页并刷新
6) 非法 cron 保存返回 422，前端阻止提交并提示
7) /config 不存在时，系统能创建默认值或返回可用默认配置
8) 后端短时异常（5xx/超时）时，前端显示可重试提示，不静默失败

三、证据要求
1) 每项测试给出：
- 结果：通过/失败
- 证据：关键请求与响应（状态码+核心字段）、关键日志、必要截图
- 失败项的根因与修复建议
2) 对 scheduler 相关项，附：
- cron 更新前后的 job 信息
- 实际触发日志（开始/结束/跳过）
3) 给出最终结论：
- 是否达到上线标准
- 阻塞问题清单（按严重级别排序）

四、输出格式（严格）
1) 概览：总用例数、通过数、失败数、阻塞数
2) 用例明细表：编号 | 场景 | 预期 | 实际 | 结果 | 证据
3) 缺陷清单：严重级别 | 现象 | 复现步骤 | 建议修复
4) 回归结论：可发布/不可发布 + 理由
