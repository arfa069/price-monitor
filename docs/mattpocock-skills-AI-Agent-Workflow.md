● 工作流程解析

  这套技能集围绕一个核心循环设计：需求 → 编码 → 反馈 → 架构维护

  ---
  完整工作流

  新任务开始
      │
      ▼
  ┌─────────────────────────┐
  │  1. /grill-me            │  ← 烤问需求，把歧义消灭在开始前
  │     或 /grill-with-docs   │
  └────────┬────────────────┘
           │ 产出: CONTEXT.md (共享术语库)
           ▼
  ┌─────────────────────────┐
  │  2. 明确需求后            │
  │     → /to-prd 生成 PRD    │
  │     → /to-issues 拆 Issue │
  └────────┬────────────────┘
           │ 分配给 AI 或自己
           ▼
  ┌─────────────────────────┐
  │  3. /tdd                 │  ← 测试先行，红绿重构循环
  │     + /diagnose          │  ← 出 Bug 就进诊断流程
  └────────┬────────────────┘
           │ 编码完成
           ▼
  ┌─────────────────────────┐
  │  4. /improve-codebase-   │  ← 定期运行，防止代码腐化
  │     architecture          │
  └────────┬────────────────┘
           │ 新任务 / Bug
           ▼
        回到步骤 1

  ---
  三条核心反馈循环

  ┌──────────────┬────────────────────────────────┬────────────────┐
  │     循环     │              工具              │    触发时机    │
  ├──────────────┼────────────────────────────────┼────────────────┤
  │ 需求对齐循环 │ /grill-me → CONTEXT.md         │ 每次新任务开始 │
  ├──────────────┼────────────────────────────────┼────────────────┤
  │ 编码质量循环 │ /tdd → 红绿重构 → /diagnose    │ 每次写代码     │
  ├──────────────┼────────────────────────────────┼────────────────┤
  │ 架构维护循环 │ /improve-codebase-architecture │ 每隔几天跑一次 │
  └──────────────┴────────────────────────────────┴────────────────┘

  ---
  共享语言机制（关键创新）

  CONTEXT.md 是整个工作流的粘合剂：

  # 共享术语
  素材化级联问题 = "当课程章节中的某课时获得实际位置时触发的问题"

  把长描述压缩成精确术语 → AI 每次理解成本降低 → token 节省约 75%

  ---
  caveman 模式（省 token 终极手段）

  当你和 AI 已经建立好共享术语库后，切到 /caveman 模式：

  - 用缩写、术语、梗沟通
  - 不解释上下文
  - AI 用同样方式回应
  - 典型场景：AI 已经开始正确干活后，微调/小改动

  ---
  本质

  这套工作流的核心洞察是：AI Agent 失败的原因 90% 都在需求对齐环节，而不是编码能力。/grill-me 就是来解决这个问题的 —— 在 AI
  开始写代码之前，先把它"拷问"清楚。
